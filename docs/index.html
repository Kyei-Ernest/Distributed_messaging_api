Distributed Concepts Demonstrated

1. Message Passing (No Shared Memory)

  • Django and Go run as separate processes
  • Communication only through Redis Pub/Sub
  • Network-based message exchange
  • Services can be deployed on different machines

2. Multicast Communication

  • Single message sent to group channel
  • All online members receive simultaneously
  • Go server routes to all active WebSocket connections

3. Unicast Communication

  • Private messages target specific user ID
  • Django validates membership
  • Only recipient's WebSocket receives message

4. Active Membership Management

  • Go tracks WebSocket connections in real-time
  • Automatic join/leave event broadcasting
  • Online member list updated dynamically

Limitations and Future Improvements

Current Limitations

  • Single Go server limits scalability (~10K connections)
  • No message ordering guarantees across clients
  • No end-to-end encryption
  • Limited fault tolerance

Future Enhancements

  • Horizontal scaling with multiple Go servers
  • Message acknowledgments and delivery receipts
  • End-to-end encryption using Signal Protocol
  • File sharing capabilities
  • Voice/video calls using WebRTC
  • Mobile applications

Conclusion

The system successfully demonstrates all required distributed systems concepts through a practical, production-ready messaging application. The modular architecture allows independent scaling and upgrading of components while maintaining clear separation of concerns.
